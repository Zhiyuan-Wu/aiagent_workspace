* Qlib Backtest Framework Analysis (2026-03-02): Deep source-level comparison between myalpha.py (Qlib's TopkDropoutStrategy) and backend (custom implementation) revealed 3 critical differences causing annual return divergence. Experimental validation completed for all hypotheses: (1) PRIMARY CAUSE CONFIRMED: Qlib calculates buy budget as `cash * risk_degree / len(buy)` WITHOUT considering open_cost (line 269-270 in signal_strategy.py is commented out), while backend uses `available_cash / len(buy)` which accounts for transaction costs. This ~95/stock difference compounds over time significantly. (2) trade_unit default: REJECTED - only 1.05% impact. Qlib uses config value C.trade_unit (possibly None), backend hardcodes to 100 (round lots). (3) forbid_all_trade_at_limit default: REJECTED - 0.00% impact. Qlib defaults True (no trading at limits), backend defaults False (one-way allowed). Key insight: When using multiple backtest frameworks, always verify parameter defaults and cost accounting logic - even "minor" differences in buy budget calculation can produce massive cumulative effects due to compounding. Systematic experimental validation is essential to prioritize which differences actually matter.

* Multi-Source Data Architecture (2026-03-03): Successfully implemented Yahoo Finance as secondary data source for alpha_mining project. Key design decisions: (1) Separate OHLCV and amount collection for indices then merge - different endpoints provide different required fields; (2) Dynamic timeout calculation based on file count and worker threads avoids both premature timeout kills and hanging tasks; (3) Enforce required field validation instead of silent approximation (removed volume ≈ amount/close derivation); (4) Source-specific timeout configurations (Yahoo 7200s vs Akshare 1800s) reflect different data volume characteristics. Lessons learned: Data service layer should be source-agnostic at API level but source-specific in implementation; proxy support essential for international data sources in China; comprehensive test coverage critical for multi-source systems to catch edge cases like missing fields or different data formats.

* When an experiment has been running for a long time without results: check the correctness of the code, run a small-scale experiment to estimate the time, or split it into multiple parallelizable experiments.
* When using PyTorch, utilize the MPS device to accelerate computations.
* Maintain a clean git repository: only commit essential content. Do not commit outdated versions, temporary test files, or intermediate log files.
* Complete tasks one by one. Add remaining tasks and items to be confirmed later to HEARTBEAT.md, clearly marking dependencies. Only when you are idle (or waiting for some tasks to finish), consider starting parallel tasks that have no dependencies and do not interfere with each other.

* Yahoo Finance API Date Parsing Fix (2026-03-03): Fixed ValueError when Yahoo Finance API returns timezone-aware datetime strings (e.g., "2026-02-12 09:30:00+08:00") during data normalization. The issue occurred because the code expected simple date format "%Y-%m-%d" but Yahoo returns datetime with time and timezone offset. Solution: Implement two-tier parsing strategy in normalize_yahoo() method - first use fast pd.to_datetime() with errors="coerce", then fallback to dateutil.parser.parse() for failed cases. Remove timezone info to ensure consistent naive datetimes throughout the pipeline. Implementation details: (1) Preserve original string values before parsing for fallback; (2) Use errors="coerce" to identify failed parses without raising exceptions; (3) Manually remove timezone info from timezone-aware datetimes using .replace(tzinfo=None) or .dt.tz_localize(None); (4) Added 2 comprehensive test cases: test_normalize_yahoo_handles_timezone_aware_datetime_strings and test_normalize_yahoo_handles_mixed_date_formats. All tests passing. Key lesson: When dealing with external APIs, always handle multiple datetime formats gracefully; never assume a single format will work consistently. Test with both timezone-aware and timezone-naive datetime strings to ensure robustness.
